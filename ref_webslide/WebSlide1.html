CAPAIAN PEMBELAJARAN PRAKTIKUM

Setelah praktikum, mahasiswa mampu:

  • Menyusun PRD yang jelas agar keluaran AI tidak berhalusinasi atau tidak koheren.
  • Memahami arsitektur decoupled antara frontend dan backend dalam SaaS modern.
  • Mengembangkan antarmuka berbasis Vite + React dan menghubungkannya dengan Supabase Cloud.

Keterampilan operasional

  • Mengantisipasi kebocoran kredensial, kegagalan Git workflow, dan kesalahan RLS.

MENGAPA PRD HARUS DITULIS SEBELUM CODING?

Tanpa PRD

AI cenderung menebak scope, mengubah arsitektur, menambah fitur di luar MVP, atau menghasilkan kode yang sulit disatukan.

Dengan PRD

AI bekerja dalam batas yang jelas: masalah, MVP, user flow, skema database, dan tech stack yang telah ditentukan.

Dampaknya

Output lebih terkendali, proses debugging lebih ringan, dan pekerjaan tim lebih mudah disinkronkan.

PRD adalah peta jalan arsitektur sebelum satu baris kode ditulis.

PROMPT SIAP PAKAI: PRD MULTI-TENANT SAAS

Gunakan prompt berikut untuk meminta AI menyusun dokumen PRD lengkap untuk aplikasi SaaS Personal Budgeting berbasis Multi-Tenancy. 

Bertindaklah sebagai Senior Technical Product Manager dan Cloud Architect yang berpengalaman dalam membangun aplikasi Multi-Tenant SaaS.

Saya ingin membangun sebuah aplikasi web MVP (Minimum Viable Product) untuk SaaS Personal Budgeting dengan model Multi-Tenancy, di mana setiap user/tenant terisolasi secara ketat dan tidak boleh melihat data satu sama lain.

Tolong buatkan dokumen Product Requirements Document (PRD) yang komprehensif dan terstruktur untuk proyek ini.

Gunakan batasan (Constraints) teknologi berikut:

1. Frontend:
Vite + React.js dengan pendekatan component-based UI.

2. Backend & Database:
Supabase Cloud (PostgreSQL) memanfaatkan Row Level Security (RLS) untuk isolasi tenant.

3. Hosting & Deployment:
Vercel dengan pipeline CI/CD otomatis.

Dokumen PRD harus mencakup bagian-bagian berikut:

1. Problem Statement & Value Proposition

- Jelaskan masalah utama yang dihadapi pengguna dalam personal budgeting.
- Jelaskan bagaimana model SaaS Multi-Tenant ini menjadi solusi yang aman dan skalabel.

2. Minimum Viable Product (MVP) Features

Batasi fitur hanya pada 3 fungsi inti berikut dan jabarkan fungsionalitas detailnya:

- Autentikasi & Registrasi Tenant
  Mendukung pemisahan workspace/user secara unik.

- Manajemen Transaksi
  Pencatatan pemasukan, pengeluaran, dan kategori.

- Ringkasan Dashboard Finansial
  Grafik ringkasan arus kas dan indikator limit anggaran.

3. Multi-Tenant User Flow Diagram Deskriptif

Petakan alur langkah demi langkah sejak pengguna mendaftar, membuat data anggaran pertama mereka, hingga keluar dari sistem, dengan penekanan pada aspek keamanan data terisolasi.

4. Database Schema PostgreSQL for Supabase

- Buatkan rancangan tabel yang dibutuhkan, misalnya profiles/users, transactions, dan categories.
- Wajib sertakan kolom penanda tenant, seperti user_id atau tenant_id, di setiap tabel transaksi untuk relasi database.
- Tentukan tipe data PostgreSQL standar, seperti uuid, text, numeric, timestamp, dan tipe data lain yang relevan untuk setiap kolom.

5. Security & RLS Blueprint Sederhana

Berikan rekomendasi logika kebijakan atau Policy Row Level Security (RLS) di Supabase yang harus diterapkan pada tabel transaksi agar data antar tenant tidak bocor.

Tuliskan PRD ini dengan gaya bahasa profesional, taktis, dan mudah dipahami oleh tim developer frontend.

KONEKSI VITE KE SUPABASE

// supabaseClient.js
import { createClient } from '@supabase/supabase-js'

const supabaseUrl = import.meta.env.VITE_SUPABASE_URL
const supabaseAnonKey = import.meta.env.VITE_SUPABASE_ANON_KEY

export const supabase = createClient(
  supabaseUrl,
  supabaseAnonKey
)

Kunci integrasi

Gunakan @supabase/supabase-js. Simpan URL dan anon key di .env, bukan langsung di kode sumber.

Pada Vercel, masukkan nilai yang sama melalui Environment Variables.

ENVIRONMENT VARIABLES: TITIK RAWAN KEAMANAN

VITE_SUPABASE_URL=https://xxxxxxxxxxxxxxxxxxxx.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

Aturan fatal

File .env tidak boleh pernah di-push ke GitHub. Kredensial yang bocor dapat membuka akses ke database.

Validasi .gitignore

Pastikan teks .env sudah ditambahkan sebelum melakukan commit pertama.

Produksi di Vercel

Salin VITE_SUPABASE_URL dan VITE_SUPABASE_ANON_KEY ke Project Settings > Environment Variables.

SUPABASE ROW LEVEL SECURITY (RLS)

Masalah umum

Tabel baru dapat menolak operasi read/write dari React dan memunculkan error 401/403 karena kebijakan keamanan belum disiapkan.

Fase pengembangan

Gunakan policy sementara seperti enable read and write access for all users untuk memastikan API berjalan.

Fase produksi

Perketat policy: users can only read and update their own document data dengan auth.uid().
